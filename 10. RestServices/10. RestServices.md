# Implementing Rest services

**REST services are one of the most encountered ways to implement communication between two apps**

**REST offers access to functionality the server exposes through endpoints a client can call**

## Using REST services to exchange data between apps

REST endpoints are as simple as **implementing a controller action mapped to an HTTP method and a path** and **an app calls this controller action through HTTP**
*Since an app exposes a service through a web protocol, this endpoint is called web service*

So basically **Spring REST endpoint is a controller action mapped to an HTTP method and path**
*It is just like Spring MVC architecture, but instead of a returned view and view dispatcher, the server **sends back in HTTP response directly what the controller actions returns***
![image](https://image2url.com/r2/default/images/1773082312571-1a220fdb-849a-41aa-a237-91d6b10c94d6.png)

There are a few downside of using REST endpoints:
* If the controller's action takes a lot of time to complete, the HTTP call to the endpoint might break the communication between the timeout
* Sending large quantity of data in one call might cause the call to timeout and break the communication
* Too many concurrent calls on n endpoint exposed by a backend component might put too much pressure on the app

It is **important** to consider **wht should happen if a call fails when implementing REST endpoint or could the data be affected by any way**

## Implementing a REST endpoint

*As said when implementing Spring REST endpoints the same mechanism is used just like Spring MVC*

Example for simple controller, the only new thing is ```@ResponseBody```
```java
// The class becomes  bean in the Spring context and Spring MVC knows this is  controller that mps its methods to specific HTTP paths
@Controller
public class HelloController {
    // Specifies the action path and HTTP method
    @GetMapping("/hello")
    // Tells the dispatcher servlet tht the controller's action doesn't return a view name but the data sent directly in the HTTP response
    @ResponseBody
    public String hello() {
        return "Hello!";
    }
}
```
But image if other methods in the controller returns something different than view and you have to duplicate ```@ResponseBody``` everywhere, to resolve that use ```@RestController```
**```@RestController``` instructs Spring that all the controller's actions are REST endpoints**
```java
@RestController
public class HelloController {
    @GetMapping("/hello")
    public String hello() {
        return "Hello!";
    }

    @GetMapping("/ciao")
    public String ciao() {
        return "Ciao!";
    }
}
```

In order to validate how do they work correctly, there are must-learn tools for a developer:
* **Postman**, *offers nice GUI and is comfortable to use* 
* **cURL**, **a command-line tool useful in cases where you dont have GUI**

## Managing the HTTP response

The HTTP response **is how the backend app sends dta back to the client due to a client's request**
The HTTP response holds data as the following:
* *Response headers*, **represents short pieces of data in the response**
* *Response body*, **A larger amount of data the backend needs to send in the response**
* *Response status*, **A short representation of the request's result**
