# Implementing Rest services

**REST services are one of the most encountered ways to implement communication between two apps**

**REST offers access to functionality the server exposes through endpoints a client can call**

## Using REST services to exchange data between apps

REST endpoints are as simple as **implementing a controller action mapped to an HTTP method and a path** and **an app calls this controller action through HTTP**
*Since an app exposes a service through a web protocol, this endpoint is called web service*

So basically **Spring REST endpoint is a controller action mapped to an HTTP method and path**
*It is just like Spring MVC architecture, but instead of a returned view and view dispatcher, the server **sends back in HTTP response directly what the controller actions returns***
![image](https://image2url.com/r2/default/images/1773082312571-1a220fdb-849a-41aa-a237-91d6b10c94d6.png)

There are a few downside of using REST endpoints:
* If the controller's action takes a lot of time to complete, the HTTP call to the endpoint might break the communication between the timeout
* Sending large quantity of data in one call might cause the call to timeout and break the communication
* Too many concurrent calls on n endpoint exposed by a backend component might put too much pressure on the app

It is **important** to consider **wht should happen if a call fails when implementing REST endpoint or could the data be affected by any way**

## Implementing a REST endpoint

*As said when implementing Spring REST endpoints the same mechanism is used just like Spring MVC*

Example for simple controller, the only new thing is ```@ResponseBody```
```java
// The class becomes  bean in the Spring context and Spring MVC knows this is  controller that mps its methods to specific HTTP paths
@Controller
public class HelloController {
    // Specifies the action path and HTTP method
    @GetMapping("/hello")
    // Tells the dispatcher servlet tht the controller's action doesn't return a view name but the data sent directly in the HTTP response
    @ResponseBody
    public String hello() {
        return "Hello!";
    }
}
```
But image if other methods in the controller returns something different than view and you have to duplicate ```@ResponseBody``` everywhere, to resolve that use ```@RestController```
**```@RestController``` instructs Spring that all the controller's actions are REST endpoints**
```java
@RestController
public class HelloController {
    @GetMapping("/hello")
    public String hello() {
        return "Hello!";
    }

    @GetMapping("/ciao")
    public String ciao() {
        return "Ciao!";
    }
}
```

In order to validate how do they work correctly, there are must-learn tools for a developer:

## Managing the HTTP response

The HTTP response **is how the backend app sends dta back to the client due to a client's request**
The HTTP response holds data as the following:
* *Response headers*, **represents short pieces of data in the response**
* *Response body*, **A larger amount of data the backend needs to send in the response**
* *Response status*, **A short representation of the request's result**

#### Sending objects as a response body

In order to send an object to the client in a response **is to make sure the controller's action return that object**

*For this example we are going to define a model object named Country*
```java
public class Country {
    private String name;
    private int population;

    public static Country of(String name, int population) {
        Country country = new Country();
        country.setName(name);
        country.setPopulation(population);
        return country;
    }

    // getters and setters
}
```
*Objects modelling data sent between two apps are named **data transfer objects (DTO)***

Example on how the controller has a controller's action that returns an object
```java
@RestController
public class CountryController {
    @GetMapping("/france")
    public Country france() {
        return Country.of("France", 67);
    }
}
```
How would this object look in the HTTP response body, well by default **Spring creates String representation of the object and formats it into JSON**
```json
{
    "name": "France",
    "population": 67
}
```

You could also send **object collection instances in the response body**
```java
@RestController
public class CountryController {
    @GetMapping("/all")
    public ListM<Country> countries() {
        Country france = Country.of("France", 67);
        Country getmany = Country.of("Germany", 88);

        return List.of(france, germany);
    }
}
```
The result would be
```json
{
    {
        "name": "France",
        "population": 67
    },
    {
        "name": "Germany",
        "population": 88
    }
}
```
*Using JSON is the most common way to represent objects when working with REST endpoints*

### Setting the response status and hearders

By default Spring sets some common HTTP statuses:
* **200 OK**, *if no exception was thrown on the server side while processing the request*
* **404 Not Found**, *if the requested resources doesnt exists*
* **400 Bad Request**, *if a part of the request could not be matched with the way the server expected the data*
* **500 Error on server**, *if an exception was thrown on the server side while processing the request*

**The easiest and most common way to customize the server's HTTP response is using ```ResponseEntity``` class**

*This class allows you to specify the response body, status and the headers of the HTTP request*

Foe example:
```java
@RestController
public class CountryController {
    @GetMapping("/france")
    // The controller's action will return response entity object of type Country
    public ResponseEntity<Country> france() {
        Country c = Country.of("France", 67);
        
        // here we are specifying what the ResponseBody will hold when it is returned as a response to the client
        return ResponseEntity
            .status(HttpStatus.Accepted)
            .header("continent", "Europe")
            .header("capital", "Paris")
            .header("favourite_food", "cheese and wine")
            .body(c);
    }
}
```

### Catching exceptions with ```@RestControllerAdvice``` 

```@RestControllerAdvice``` represetns **an aspect that intercepts exceptions thrown by controller's action and define yourself logic for catching the exception**

Here is a picture of how the general architecture would look like
![image](https://image2url.com/r2/default/images/1773222104585-701b2339-9401-4359-86ad-e393583fb051.png)

Here is how the controller would look like now, it is much more simplified
```java
@RestController
public class PaymentController {

    @Autowired
    private final PaymentService paymentService;

    @PostMapping("/payment")
    public ResponseEntity<PaymentDetails> makePayment() {
        PaymentDetails paymentDetails = paymentService.processPayment();
        return ResponseEntity
            .status(HttpStatus.ACCEPTED)
            .body(paymentDetails);
    }
}
```

But now how we will catch ```NotEnoughMoneyException``` if its thrown? Well, we will create ```ExceptionControllerAdvice``` class that is a ***REST controller advice***

In order to mark a class being a ***REST controller advice***, we use ```@RestControllerAdvice``` annotation, the method the class defines is also called ***exception handler***

In order to specify ***what exception will trigher the controller advice method we use*** ```@ExceptionHandler``` annotation over the method
```java
@RestControllerAdvice
public class ExceptionControllerAdvice {
    // this method is going to be executed the method NotEnoughMoneyException is going to be called from somewhere
    @ExceptionHandler(NothEnoughMoneyException.class)
    public ResponseEntity<ErrorDetails> exceptionNoEnoughMoneyHandler() {
        ErrorDetails errorDetails = new ErrorDetails();
        errorDetails.setMessage("Not enough money to make the payment");
        return ResponseEntity
            .badRequest()
            .body(errorDetails);
    }
}
```

## Using a request body to get data from the client

*Since the HTTP request has a body **that body can be used to send data from the client to the server***

*Remember, the http body is often used with Rest endpoints and **its primarly used when sending large amount of data***

If you want to use the request body, you must use ```@RequestBody``` annotation to annotate a parameter of the controller's action.

*By default Spring assumes you used JSON to represent the parameter and will decode the JSON string to instance the parameter, if he cant decode it it will return 400 Bad Request*

Here's an example on how the controller's action will accept the client's request body
```java
@Controller
public class PaymentController {
    @PostMapping("/payment")
    // the passed client's body will be converted to PaymentDetails object becasue of @RequestBody
    public ResponseEntity<PaymentDetails> makePayment(@RequestBodu PaymentDetails paymentDetails) {
        return ResponseEntity
            .status(HttpStatus.ACCEPTED)
            .body(paymentDetails);
    }
}
```
