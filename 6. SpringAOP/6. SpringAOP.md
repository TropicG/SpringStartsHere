# Using aspects with Spring AOP

**Aspects are a way the framework intercepts method calls and possibly alters the execution of methods.**

## How aspects work in Spring

**An aspect is simply a piece of logic the framework executes when you call specific methods of your choice**

When designing an aspect, you define the following
* What code you want Spring to execute when you call specific methods, this is named **aspect**
* When the app should execute the logic of the aspect, this is named **the advice**
* Which methods the framework needs to intercept and executre the aspect for them, this is named **a pointcut**

*You will also find the concept of **join point** which defines the event that triggers the execution of the aspect*

In order to use the aspect **you need the framework to manage the objects for which you want to apply aspects**

The bean that declares the method intercepted by an aspect is named **the target object**

![](https://image2url.com/r2/default/images/1771844197848-e41cc3e6-19b5-4a66-b89c-8abe5ef2f4d4.png)

When one bean is the **aspect target**, Spring **won't directly give you an instance reference for the bean** when it is requested from the context

Instead **Spring gives you an object that calls the aspect logic instead fo the actual method**, we can say that Spring **gives you a proxy object instead of the real bean**

*This proxy object can manage each call to the intercepted method, it is an object that enhances the behavior of a class*

Calling an aspected method assumes you call the method **through the proxy provided by Spring**, the proxy **applies the aspect logic and delegates to call the actual method**

![](https://image2url.com/r2/default/images/1771845849704-b52a1c1a-73df-4846-8f95-07de1772363a.png)

## Implementing aspects with Spring AOP

*In this section we will create a new project and dfine a service class containing a method that create an aspect for it*

*For the following project in addition to adding the spring-context depedency, you have to add the spring-aspects depedency*

In order to be more focused on the aspects, we will create one service object CommentService and its method publicComment(Comment comment), which accepts Comment which is a model class

```java
package model;

public class Comment {
    private String text;
    private String author;

    // getters and setters
}
```
```java
package service;

@Service
public class CommentService {
    // In real-world apps logging frameworks are used, just dont give much attention to this data member
    private Logger logger = Logger.getLogget(CommentService.class.getName());

    public void publishComment(Comment comment) {            
        logger.info("Publishing comment:" + comment.getText());
    }
}
```

Dont forget for the ProjectConfig class
```java
@Configuration
@ComponentScan(basePackages = "services")
public class ProjectConfig{}
```

We will now enhance the project with an aspect class that intercepts the method call and adds an output before and after the call, we have to do the following steps:
* Enable the aspect mechanism in Spring **by annotating the configuration class with** ```@EnableAspectJAutoProxy``` annotation
* Create a new class **and annotation it with the** ```@Aspect``` annotation, *adding either with ```@Bean``` or stereotype annotations, add a bean for this class in the Spring context*
* Define a method that will implement the aspect logic and tell Spring when and which methods to intercept using an advice annotation
* Implement the aspect logic 

The configuration class will look like this in order to enable the aspect annotations
```java
@Configuration
@ComponentScan(basePackages = "services")
@EnableAspectJAutoProxy
public class ProjectConfig{}
```

Lets create a bean that will define the aspects, this object **holds the methods, which will intercept specific method calls and augment them with specific logic**
```java
@Component
@Aspect
public class LoggingAspect {
    @Around("execution(* services.*.*(..))")            
    public void log(ProceedingJoinPoint joinPoint) {
        // prints a message to the console before the intercepted method's execution   
        logger.info("Method will execute");
        // calls the intercepted methods
        joinPoint.proceed();            
        // prints a message in the console after the intercepted method's execution   
        logger.info("Method executed");               
    }
}
```
*Remember, the @Aspect annotations doesnt tell Spring to create a bean for this class, it tells Spring that this class implements definition of Aspect*

The ```@Around``` annotation is used to **tell spring when and which method calls to intercept**, and the expression as a parameter **tells Spring which method calls to intercept**

Although the expression looks very complex take a look at this picture:
![](https://image2url.com/r2/default/images/1771946080248-ca32c586-f598-400e-bbe1-c370251171ad.png)

In the metod you could find the ProceedingJoinPoint parameter *(it represents the intercepted method)*. This parameter is used to tell the aspect **when it should delegate further to the actual method**

This is the overall picture on how the actual interception of the method is happening
![](https://image2url.com/r2/default/images/1772004281644-73da8768-4b97-4d64-aa3a-a1491b46da0c.png)

The method **proceed()** actually calls the intercepted method, if you do not call it **the aspect will never delegates further**

**proceed() throws Throwable**, this means that it will **throw any exception coming from the intercepted method**

Aspects even have the ability to **intercept the parameters used to call the method and possibly alter them or the value the intercepted method returns**
As you know **ProceedingJoinPoint represents the intercepted method**, in this parameter we can find any information for the intercepted method, for example:
* joinPoint.getSignature().getName(), **this will return the name of the intercepted method**
* Object[] arguments = joinPoint.getArgs(), **this will get all the arguments passed to the method**
 
Aspects are indeed so poowerful, they can alter different parts of the method:
* Changing the value of the parameters sent to the method
* Changing the returned value received by the caller
* Throwing an exception to the caller or catching and treating an exception thrown by the intercepted method
